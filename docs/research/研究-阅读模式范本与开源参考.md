# 阅读模式范本与开源参考（研究备忘）

> 目标：网页提取以各主流浏览器「阅读模式 / Reader View」的体验与覆盖面为范本；当前实现仍以通用算法 + 轻量启发式为主，需分阶段对照差距、借力成熟开源方案迭代。  
> 工程现状摘要：`desktop/electron/webContentExtractor.ts` 使用 **`@mozilla/readability` + `jsdom`**，辅以 JSON-LD/meta、启发式 fallback、阻断页检测等；采集窗口内另用页内脚本做目录与站点 adapter（见 `desktop/electron/main.ts`）。

## 1. 范本：各浏览器阅读模式在做什么（抽象能力）

主流阅读模式共性可归纳为四层，可作为我们迭代时的「能力清单」：

| 层次 | 能力 | 说明 |
|------|------|------|
| **渲染与时机** | 在真实文档环境中等待可交互内容 | 懒加载、异步正文、软导航 SPA；需在「完整 DOM」上抽取，而不是静态 HTML 快照 |
| **主文识别** | 稳健的文章根节点与边界 | 去导航、侧栏、评论、推荐流；处理分页、「继续阅读」 |
| **可读性预判** | 是否适合进入阅读模式 | Firefox Readability 提供 `Readability-readerable.js` 等思路，避免目录页/列表页误判 |
| **体验与兜底** | 排版、字体、站点特例 | 浏览器内置大量产品与工程投入；开源侧往往用 **规则库 / 站点定制** 逼近 |

我们的差距主要在：**主文识别仍单一路径为主**、**站点/版式特例覆盖有限**、**与 Chrome/Safari 商业产品级的规则密度不可比**——这是预期内的，需按优先级逐步补规则与第二条通用引擎对比，而不是一次到位。

## 2. 与「阅读模式」直接相关的开源代码（优先参考）

下列项目与浏览器阅读视图同源或可视为「同一问题」的工程化答案，适合读源码、抄思路、评估是否引入为第二 extractor 或规则层。

### 2.1 Mozilla Readability（Firefox Reader View 核心算法）

- **仓库**：https://github.com/mozilla/readability  
- **npm**：`@mozilla/readability`（本项目已采用）  
- **要点**：对 DOM 做评分与剪枝，输出标题、正文 HTML、元数据等；**解析时会修改传入的文档**（需 clone 再 parse 若在renderer 复用同一 DOM）。  
- **附加**：`Readability-readerable.js` 用于「是否像可读文章」的轻量预判，可减少目录页误入。  
- **结论**：继续作为**默认主引擎**最合适；迭代重点放在 **前置预判、后置清洗、失败时的第二意见**。

### 2.2 Chromium DOM Distiller（Chrome 阅读相关能力的历史核心，Java 为主）

- **镜像**：https://github.com/chromium/dom-distiller  
- **官方指向**：https://chromium.googlesource.com/chromium/dom-distiller  
- **要点**：面向「文章类页面」蒸馏正文；README 注明与 Boilerpipe 等研究 lineage 相关。**仓库以 Java 为主**，与当前 Electron+Node 栈集成成本较高，更适合**读算法与启发式**（块密度、标签先验等），而非直接 npm 依赖。  
- **结论**：作**对照阅读与论文/注释级参考**；若将来要做「双引擎投票」，更可行的是再选一条 **JS 生态** 流水线（见下）而非硬嵌 DOM Distiller。

### 2.3 Postlight Mercury Parser / 社区维护分支

- **原仓库**：https://github.com/postlight/parser（又名 mercury-parser）  
- **维护分叉示例**：https://github.com/jocmp/mercury-parser（社区延续）  
- **要点**：**通用抽取 + 可扩展的 per-host 自定义 extractor**（CSS 选择器/定制脚本），与「阅读模式 + 大厂站点补丁」的产品形态接近。  
- **结论**：适合作为 **「规则层 + 第二通用引擎」** 的参考；接入需评估包体积、许可、依赖活跃度及与 Readability 结果 **一致性/仲裁策略**。

### 2.4 WebToEpub（扩展：海量站点 Parser + Default Parser）

- **仓库**：https://github.com/dteviot/WebToEpub  
- **要点**：优势不在单一算法，而在 **大量站点专用 parser + 默认解析器 + 用户可改模板**，与「长线覆盖」路线一致。  
- **结论**：研究其 **DefaultParser / Template** 与各站 parser 的**组织方式**，为我们自己的 **adapter 注册表**提供工程范本。

### 2.5 其它常被阅读类产品采纳的栈（偏产品与整合）

| 项目 | 链接 | 参考价值 |
|------|------|----------|
| **Omnivore** | https://github.com/omnivore-app/omnivore | Readability 类方案与自研/规则混合的产品级整合 |
| **ArchiveBox** | https://github.com/ArchiveBox/ArchiveBox | 归档链路中与正文抽取的配合 |
| **SimpRead（产品向）** | 以站点适配与阅读体验著称 | 规则层与交互设计参考（实现未必适合直接依赖） |

### 2.6 Safari 相关

- WebKit/Safari Reader **完整源码不在一小个 MIT 仓库里**，网络上存在**第三方整理的镜像或逆向片段**，版权与可追溯性参差不齐，**仅适合概念对照**，不宜当「可合并上游」。  
- 若需 Apple 生态行为对齐，仍以 **自测页面集 + Readability/自研规则** 为准。

### 2.7 非 JS 但可借鉴「新闻正文」启发式

- **Newspaper3k**（Python）：https://github.com/codelucas/newspaper — 统计与启发式，可作**论文与特征**参考，不适合直接并进 Electron 主进程。  
- **go-readability** 等端口：同一算法的跨语言实现，可用于**离线对比测试**（与同页 HTML 输入对齐）。

## 3. 与我们当前实现的映射

| 范本能力 | 当前工程大致对应 | 后续可增强方向 |
|----------|------------------|----------------|
| 真实 DOM 渲染 | `BrowserWindow` + `executeJavaScript` 取 HTML | 更细的等待条件、登录后重试、限流与超时策略 |
| 主文抽取 | Readability + structured/meta + `fallbackExtract` | 引入 **readerable 预判**；双引擎或 Mercury 式 **host 规则** |
| 目录/小说站 | `web:extractBookDetail` 页内脚本 + 少量 adapter | **adapter 注册表**、与 WebToEpub 类似的默认解析路径 |
| 反爬/乱码/iframe | `ExtractError` 多类检测 | 专项迭代（字体混淆、跨域 iframe 提示与绕过指引） |
| 手动兜底 | `web:extractFromSelection` | 与 UI 强结合的「框选即保存」体验 |

## 4. 研究结论与建议迭代顺序（务实路线）

1. **保留 Readability 为主线**：与 Firefox 阅读模式一致，社区与文档成熟，已证明可嵌入 Electron。  
2. **先做「失败可解释 + 预判」**：借鉴 `Readability-readerable.js` 思路，减少目录页进阅读流；错误码已与产品提示挂钩，可继续细化。  
3. **再做「规则层」**：按 hostname 注册轻量 selector / 预处理（参考 Mercury custom extractors、WebToEpb parser 列表），**不必**自研第二套大算法。  
4. **可选「第二通用引擎」**：在 `<200 字符` 或 Readability 与启发式分歧大时，用 Mercury 类结果做 **仲裁或合并**（需定义主从策略，避免正文重复或截断）。  
5. **Chrome DOM Distiller**：以 **阅读源码与论文脉络** 为主，除非团队能接受 JVM/ JNI 桥，否则不急于工程集成。  
6. **评测集驱动**：收集「用户真实失败 URL」（脱敏）形成固定夹具，每次改提取链跑回归——这是追赶浏览器阅读模式覆盖面的唯一可持续办法。
