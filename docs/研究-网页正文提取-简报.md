# 网页正文提取技术调研简报（面向“最大通用兜底”）

## 1) 参考项目核心提取逻辑与兜底策略

### A. WebToEpub（小说站适配非常强）
- **核心思路**
  - 以“站点专用 parser”为主：每个站点实现 `getChapterUrls`、`findContent`、`extractTitle` 等。
  - 提供模板 parser，允许快速复制改造，适配奇怪 DOM 结构（比如标题和正文不在同容器、目录分页等）。
- **兜底策略**
  - 当无站点 parser 时使用 `DefaultParser`，让用户提供 CSS selector（正文、章节标题、清理规则）。
  - 可手动切换 parser（FAQ 中 Baka-Tsuki 的例子就是自动识别不稳定时人工兜底）。
  - 支持请求节流/最小间隔（`minimumThrottle`）及对 429 场景的降速建议。
- **对非标准网站**
  - 论坛贴文/个人博客可通过 selector 指向正文容器；必要时移除广告、分享块。
  - 本质上是“规则驱动 + 人工可修正”的工程化路线。

### B. Mercury Parser / Postlight Parser（规则+通用并存）
- **核心思路**
  - `Parser.parse()` 先挑选站点 extractor（custom），否则走 generic extractor。
  - custom extractor 通过 selectors、clean、transforms 精准提取；适合模板稳定站点。
  - generic extractor 会对最佳节点进行打分与清洗。
- **兜底策略**
  - custom selector 未命中时，回退到 generic extractor。
  - generic extractor 内部做“逐步放宽”重试：`stripUnlikelyCandidates`、`weightNodes`、`cleanConditionally` 依次关闭再试。
  - `fallback` 参数可控制是否允许回退（测试时常用 `fallback: false` 验证规则）。
- **可扩展性**
  - 支持运行时 `addExtractor()` / CLI `--add-extractor`。
  - 支持 `extend` 字段抽取（补充业务字段，而不改核心协议）。

### C. Newspaper3k（启发式与文本统计）
- **核心思路**
  - 继承 goose 风格：候选节点筛选 + stopword 统计 + link density 惩罚 + parent/grandparent 传播加权。
  - `calculate_best_node()` 里使用 stopword 数量作为主信号，结合高链接密度过滤。
  - 对 sibling 段落做并入，提升正文连续性。
- **兜底策略**
  - 标题、作者、发布时间采用多路策略（URL、meta、多种属性匹配）。
  - 非英语通过不同语言 stopwords/tokenization 子类适配（中文、日文等）。
- **对非标准网站**
  - 纯启发式对“结构脏但文本量大”页面有韧性；
  - 但对短章节、强模板噪声站点可能需要规则补刀。

### D. Firefox / Mozilla Readability（通用算法基准）
- **核心思路**
  - 使用 class/id 正负权重、段落长度、逗号/文本特征、链接密度、可见性检测等构建候选评分。
  - 预判函数 `isProbablyReaderable()` 用低成本规则决定是否值得跑完整解析。
  - 解析时支持 JSON-LD 元数据优先（可关闭），并进行大量清洗（表单、侧栏、分享组件等）。
- **兜底策略**
  - 失败时会在 flags 层面迭代重试（剥离 unlikely、权重、条件清洗等阶段化）。
  - 具备较强通用性，适合做第一层引擎。

### E. Safari Reader（WebKit ReaderArticleFinder）
- **核心思路**
  - 候选块几何约束（宽高/面积/位置）+ 文本节点打分 + 语言倍率（CJK）+ 密度判定。
  - 对标题相似性、注释区排除、并行结构干扰有专门规则。
  - `nextPageURL` 推断逻辑较强：链接语义（next/page/continue）、URL 相似度、位置权重等综合打分。
- **兜底策略**
  - 白名单优先（trusted selector）→ 全元素搜索 → 视觉检查路径。
  - 对多页文章有拼接机制（不是只抽当前容器）。

### F. 单行阅读类工具（OneLine/Brutal Reader/ripcurl 类）
- **核心观察**
  - 多数“单行阅读”产品重心在展示层（聚焦高亮、视窗遮罩），正文抽取能力通常不深。
  - 开源里 `Brutal Reader` 更像轻量 reader mode；`ripcurl` 偏 CLI 文本清洗与反 Cloudflare 基础绕行。
- **可借鉴点**
  - UI 与抽取解耦：抽取失败可切“纯文本模式”或“用户手选容器模式”。
  - 给用户可见的降级体验，比“算法必须一次成功”更实用。

## 2) 针对调研重点的横向结论

### 非标准网站（论坛、个人博客）
- 最稳方案不是单算法，而是 **三层组合**：
  1) 通用算法（Readability/Mercury generic）  
  2) 站点规则（custom extractor / parser）  
  3) 人工 selector 兜底（WebToEpub DefaultParser 模式）

### 导航识别（上一章/下一章）
- 高优先级：`rel="next"/"prev"`、明确锚文本。
- 文本不明确时，借鉴 Safari：
  - URL 结构相似度（同路径骨架、末尾数字 +/-1）；
  - 与正文容器的空间距离（近正文尾部更可信）；
  - query 参数语义（`page`, `p`, `pg`）。
- 目录缺失时，可先“单章入库 + 邻接链接扩展”。

### 反爬 / 动态渲染
- 这类项目主流不做强对抗；通常依赖“浏览器真实渲染后再抽取”。
- 实践上建议：
  - 在 Electron `webContents` 中等待就绪 + 增量滚动触发懒加载；
  - 复用登录态 cookie；
  - 检测验证码/登录/付费提示并返回明确错误码，而非硬抓。

### 兜底机制（最后手段）
- WebToEpub 最强点：允许用户直接配置 selector 并测试结果。
- Mercury/Readability 最强点：算法内部多轮放宽重试。
- 工程上最终兜底应包含：**手动框选容器 + 纯文本导入**。

## 3) 对你们项目的 3 条具体改进建议

- **建议 1：实现“JSON-LD / Meta 优先的元数据前置层”**
  - 在主提取前先抓 `script[type="application/ld+json"]`、`og:title`、`article:published_time`、`author`。
  - 命中后可作为标题/作者/时间的稳定锚点，减少正文误提取带来的字段污染。
  - 与 Readability 的 `disableJSONLD=false` 思路一致。

- **建议 2：落地“分阶段退化链（可观测）”**
  - 推荐链路：  
    `Readability(默认) -> Readability(放宽参数) -> Mercury generic风格密度算法 -> 站点规则 -> 手动框选`
  - 每层都记录失败原因（长度不足、链接密度过高、无候选节点、命中登录墙等），便于后续策略迭代。
  - UI 上展示当前阶段与下一步建议，降低“黑盒失败”体验。

- **建议 3：独立“章节导航推断器”并引入 URL 结构评分**
  - 组合特征：锚文本词典 + `rel=next/prev` + URL 数字序列 + 与正文区域距离 + 域名/路径相似度。
  - 当无明确“下一章”文案时，用 URL 结构推断候选并打分，低置信度只提示不自动跳转。
  - 先做保守自动化：宁可少跳，不要误跳到评论页/推荐页。

## 4) 建议的实现优先级（短期）
- P0: 增加 JSON-LD/Meta 前置抽取 + 错误码可观测化。
- P1: 完成 Readability 两段式重试 + 密度算法兜底（参考 Newspaper/Mercury generic）。
- P2: 上下章识别器（URL 结构评分）+ 手动框选容器保存。
