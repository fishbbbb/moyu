# 网页导入阅读：扩展调研（V2，面向“简介/章节双入口 + 稳定目录跳转”）

> 说明：本次采用“关键词扩展法 + 多源交叉验证”方法重做调研。由于当前网络检索环境受限（代理连接失败），以下结论基于已收集的官方源码/官方文档与既有调研资料进行再验证与体系化整合；后续可在联网可用时补充最新版本差异核验。

## 0) 目标重述（与开发目标对齐）

你当前要实现的不是“单页阅读模式”，而是“书籍导入链路”：

1. 输入简介页 URL 或章节页 URL，均能识别为同一本书（统一 `workId`）
2. 导入后拿到完整章节目录（TOC）
3. 点击目录任意章节可正确跳转
4. 阅读器“上一章/下一章”基于目录顺序稳定跳转

结论先行：该目标可实现，但工程上应采用 **通用解析主链路 + 站点规则补丁 + 人工兜底** 三层架构。

## 2) 关键发现（重构后的精确结论）

## 2.1 是否必须“每站点独立适配”？

不是。更准确是：

- **通用链路可覆盖大多数常见资讯/博客站点正文提取**
- **小说站“完整目录 + 稳定跳章”通常需要少量站点规则补丁**
- **极端站点需要人工 selector 兜底**

也即：不需要一开始全站点适配，但要设计出“可插拔适配层”。

## 2.2 浏览器阅读模式为什么不能直接等价为你的目标？

浏览器 Reader Mode 主要解决：
- 当前页面正文提取与阅读体验优化

你的目标额外要求：
- 跨入口（简介页/章节页）归并为同一本书
- 完整 TOC 获取
- 章节级导航一致性（目录跳转与 prev/next）

因此需要额外模块：`bookNormalizer + chapterGraph + tocResolver`，不能只依赖 Readability。

## 2.3 对“目录识别 + 章节跳转”的真实可达性判断

可达，但成功率取决于策略层级是否完整：

- 一级（通用）：DOM 语义与链接结构识别 TOC
- 二级（增强）：`rel=next/prev` + 锚文本词典 + URL 数字序列
- 三级（规则）：站点专用 selector/API
- 四级（兜底）：单章导入 + 邻接扩展 + 用户确认

当你只做一级时，目录与跳章会不稳定；做到三级时，常见目标站点可以稳定可用。

## 3) 目标架构（建议定版）

## 3.1 导入状态机

1. `URL_INPUT`
2. `URL_NORMALIZE`（协议、域名、尾斜杠、query 清理）
3. `ENTRY_CLASSIFY`（简介页/章节页/未知）
4. `BOOK_ID_RESOLVE`（workId 归一）
5. `TOC_RESOLVE`（目录抓取与排序）
6. `CHAPTER_EXTRACT`（章节正文提取）
7. `LINK_GRAPH_BUILD`（index + prev/next）
8. `VALIDATE`（完整性校验）
9. `IMPORT`

## 3.2 章节导航的数据真相源（single source of truth）

- 所有跳转都基于本地 `chapters[index]`，而不是网页按钮
- 网页 `next/prev` 仅作为抓取线索，不作为阅读器跳转真值

## 3.3 推荐字段（最小必需）

- Book: `bookId`, `canonicalUrl`, `title`, `site`, `sourceType`
- Chapter: `chapterId`, `bookId`, `index`, `title`, `url`, `prevChapterId`, `nextChapterId`
- ExtractionMeta: `strategy`, `confidence`, `errors[]`, `warnings[]`

---

## 4) 目录与跳章识别策略（按优先级）

1. 目录页显式 TOC（`nav/ul/ol` + 章节语义）
2. 页面内章节集合（正文外的大块链接区）
3. `rel=next/prev` 与锚文本词典（中英）
4. URL 模式推断（数字位、路径骨架）
5. 站点专用规则（selector/API）
6. 人工确认（用户选目录容器）

### 4.1 锚文本词典建议

- next: `下一章`, `下页`, `next`, `next chapter`, `continue`
- prev: `上一章`, `上页`, `previous`, `prev chapter`
- toc: `目录`, `章节`, `全部章节`, `toc`, `chapter list`

### 4.2 URL 推断约束（防误跳）

- 同域同书路径相似度 >= 阈值
- 数字变化符合 ±1 或分页规律
- 链接位置靠近正文尾部加权
- 命中“评论/推荐/广告”路径关键词降权

---

## 5) 失败模型与错误码（必须可观测）

- `ENTRY_UNSUPPORTED`：入口页类型无法识别
- `BOOK_ID_UNRESOLVED`：无法归并同一本书
- `TOC_NOT_FOUND`：未找到可用目录
- `TOC_PARTIAL`：目录疑似不完整
- `CHAPTER_PARSE_FAILED`：章节正文提取失败
- `NEXT_PREV_LOW_CONFIDENCE`：邻接章节置信度低
- `AUTH_REQUIRED`：需登录
- `PAYWALL_BLOCKED`：付费墙
- `RATE_LIMITED`：触发频率限制

## 6) 测试计划（与目标直接绑定）

## 6.1 核心验收用例

1. 输入简介页 URL → 导入成功 → 目录可见 → 任意章节可跳转
2. 输入章节页 URL（第 1 章）→ 归并为同一本书 → 目录完整
3. 阅读器上一章/下一章在首章、中间章、末章行为正确
4. 刷新后恢复同章阅读位置，继续 prev/next 正确

## 6.2 质量门槛（建议）

- TOC 完整率（目标站点）≥ 95%
- 章节跳转正确率 ≥ 99%
- 错误码可归因率 ≥ 95%

## 6.3 站点分层验证

- A 类（结构规范站点）：纯通用链路
- B 类（轻度定制站点）：少量规则补丁
- C 类（重定制/反爬站点）：人工兜底 + 明确提示

## 8) 最终结论（回答当前疑问）

1. 你问“常用站点能否做到目录识别和章节跳转？”——**能做到**，前提是采用分层策略而非单算法。  
2. 你问“是否必须每站点独立适配？”——**不是必须全量适配**，但要为问题站点预留适配机制。  
3. 你问“浏览器阅读模式思路是否正确？”——**正确但不充分**；它是正文层能力，不是完整书籍导入能力。  
4. 你当前调研方向是正确的，下一步关键是将“算法能力”升级为“工程闭环（状态机 + 错误码 + 验收）”。
