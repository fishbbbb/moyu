# 网页导入阅读：技术调研与集成指南（整合版）

> 本文已整合《研究-网页导入阅读-扩展调研v2.md》的结论，目标是形成可直接执行的开发计划。

## 0) 目标与范围

本功能目标不是“单页阅读模式”，而是“书籍级网页导入”：

1. 用户输入**简介页 URL**或**章节页 URL**，都能归并为同一本书（统一 `bookId/workId`）
2. 可提取完整章节目录（TOC）
3. 点击目录任意章节可正确跳转
4. 阅读器内“上一章/下一章”稳定正确

结论：目标可实现，推荐采用 **通用算法主链路 + 站点规则补丁 + 人工兜底** 的三层架构。

## 1) 正文提取算法对比

### Readability（推荐主引擎）
- 库：`@mozilla/readability`
- 维护状态：活跃维护，生态成熟，常用于阅读模式与存档系统。
- 优点：
  - 对新闻/博客/资讯类页面稳定性高
  - 提取结果结构化（标题、内容 HTML、摘要、作者线索）
  - 在 Electron + DOM 环境集成成本低
- 缺点：
  - 对强交互 SPA、短内容页、目录页会误判
  - 对反阅读模式（拆 span、零宽字符）需要二次清洗

### 替代方案
- Mercury/Postlight Parser（规则能力强，维护活跃度与主线适配需评估）
- 自研启发式抽取（建议作为 fallback，不替代 Readability）

### 建议
- 主引擎：`@mozilla/readability`
- 失败或正文过短时进入 fallback（`article/main/content` 候选评分 + 链接密度约束）

## 2) 动态页面渲染（SPA）

### Electron 原生方案（推荐）
- 使用 `BrowserWindow` / `webContents` 加载目标 URL
- `webContents.executeJavaScript` 在页面上下文执行 DOM 读取
- 支持：等待 selector、滚动触发懒加载、超时控制

### 是否需要 Puppeteer
- 1.0 不建议引入：
  - 额外 Chromium 依赖导致打包体积增大
  - 进程与资源管理复杂度更高
- 2.0 若要批处理/无界面任务队列，再评估引入

---

## 3) 入口归一与书籍识别（新增重点）

### 3.1 入口分类
- 简介页（book/novel/detail）
- 章节页（chapter/read/content）
- 未知页（进入降级链路）

### 3.2 URL 归一化
- 统一协议与主域名
- 处理尾斜杠
- 去除无关 query（追踪参数）
- 记录 canonical URL

### 3.3 书籍归并（`bookId/workId`）
- 优先：页面结构化元数据（JSON-LD/meta）
- 次优：标题 + 域名 + 路径骨架
- 兜底：标题相似度去重（`fast-levenshtein`）

---

## 4) 章节目录识别与跳转策略（升级）

优先级：
1. 显式 TOC 区域（`nav/ul/ol` + 目录/章节语义）
2. 目录页中的批量章节链接区
3. `link/a[rel=next|prev]`
4. 锚文本词典（上一章/下一章/previous/next）
5. URL 数字模式推断（如 `/chapter/12 -> /chapter/13`）
6. 站点专用规则（selector/API）
7. 人工兜底（用户选择目录容器）

### 跳转正确性的核心原则
- 阅读器跳转的真值来自本地目录索引 `chapters[index]`
- 网页上的 prev/next 只作为抓取线索，不直接作为阅读器跳转依据

## 6) 登录态与付费态处理

- 登录态复用：同一 `session` cookies
- 检测策略：
  - 登录关键词：`请登录 / sign in / login required`
  - 付费关键词：`订阅后可读 / members only / subscribe to read`
- 原则：仅提示，不绕过

## 7) 失败可观测与错误码（新增）

建议统一错误码：
- `ENTRY_UNSUPPORTED`
- `BOOK_ID_UNRESOLVED`
- `TOC_NOT_FOUND`
- `TOC_PARTIAL`
- `CHAPTER_PARSE_FAILED`
- `NEXT_PREV_LOW_CONFIDENCE`
- `AUTH_REQUIRED`
- `PAYWALL_BLOCKED`
- `RATE_LIMITED`

要求：每次失败记录 `strategy + confidence + reason`，用于后续站点适配优先级排序。

## 8) Electron 安全集成建议

- 主进程负责抓取与提取，渲染层仅调 IPC
- 通过 `preload` 暴露最小必要 API（已有 `webExtractStructuredAtUrl`）
- 长操作（滚动、等待、提取）在主进程执行，避免 UI 卡顿

---

## 9) 开源参考结论（更新）

- Readability：适合作为正文主引擎
- WebToEpub：证明“站点规则 + default parser + 人工修正”对小说场景有效
- Mercury/Postlight：证明“custom extractor + generic fallback”架构可行
- Safari Reader：在 next page / 候选链接评分上有高价值启发
- Omnivore / ArchiveBox / SimpRead：证明“算法 + 规则层 + 产品兜底”是可持续路线

## 12) 在主窗口中的集成步骤

1. 打开网页：`window.api.webOpen({ url })`
2. 提取结构化内容：`window.api.webExtractStructuredAtUrl({ url })`
3. 预览：`content.textContent` / `chapters`
4. 保存：调用 `libraryImportWebItem` 或 `libraryImportWebBook`

`MainView` 中 `ensureItemHasContent()` 已使用结构化接口。

## 14) 验收测试计划（升级）

### 14.1 核心验收用例
1. 输入简介页 URL，导入后目录可见，点击任意章节可跳转
2. 输入章节页 URL（如第一章），可归并同一本书并拿到目录
3. 在首章/中间章/末章验证上一章下一章行为
4. 刷新后恢复阅读位置并继续正确跳转

### 14.2 质量门槛（建议）
- TOC 完整率（目标站点）≥ 95%
- 章节跳转正确率 ≥ 99%
- 错误码可归因率 ≥ 95%

### 14.3 站点分层
- A 类：结构规范站点（通用链路）
- B 类：轻度定制站点（少量规则补丁）
- C 类：重定制站点（人工兜底 + 清晰提示）

## 17) 当前决策结论

1. 常用站点可做到目录识别和章节跳转，但需要分层策略，不是单算法。  
2. 不需要起步就每站点适配；应先通用覆盖，再按失败热点补站点规则。  
3. 浏览器阅读模式路线正确，但仅覆盖正文提取，不等价于“书籍级导入”。  
4. 下一步重点是工程闭环：状态机、错误码、验收标准与站点分层策略。

## 18) 《网页提取与书架规范》联动：调研结论与实施状态（2026-04-12）

> 对应专题 PRD：`docs/PRD-网页提取与书架规范.md` §6（T1–T4）。本节记录**阶段性结论**、已落地行为与仍开放项，供总 PRD 与实施手册引用。  
> **总 PRD 摘要位置**：`docs/总PRD.md` §7.2.10；`desktop/docs/总PRD.md` §7.2.13（与前者对齐）。

### 18.1 T1 — 书名来源优先级与「标题置信度」

**阶段性结论**

- **桌面端书名采集**（`web:extractBookDetail` 页内脚本）：顺序为 **`h1` → `og:title` → `document.title`**；每一层结果经 `tryClean`：优先截取 **`《…》`** 书名号对；否则去掉 **`【…】`** 块、取下划线 **`_`** 之前的首段、弱化尾部「`*文学城`」类站点后缀。
- **入库前二次规范化**：主进程 `sanitizeWebBookShelfTitle`（`desktop/electron/webBookDisplay.ts`）与页内意图对齐，避免仅依赖渲染层传参再次带入噪声。
- **单页正文提取**（`web:extract` / `web:extractAtUrl`、手动框选、结构化单章拉取）返回的展示标题同样经该函数处理，保证「保存到书架」与书籍模式一致。

**仍开放**

- 是否在 IPC 返回体中增加 `titleRaw` + `titleConfidence`（0～1）写入调试或 `ExtractionMeta`，便于按失败样本排序优化；当前为隐式规则（能稳定解析 `《》` 则视为高置信）。

### 18.2 T2 — 来源展示与 `sourceLabel`

**阶段性结论**

- **1.0 采用**：`books.title` 仅存**作品本名**；**来源**以现有 **`books.domain`**（hostname）在书架列表**副标题**展示（`MainView` `toolBookSub`），**不**拼进书名。

**仍开放**

- 是否在 `books` 表增加 **`sourceLabel`**（人类可读，如「晋江文学城」），或维护 **`domain → 品牌名`** 静态映射，避免用户只看到 `www.jjwxc.net`；需与本地化、多站点扩展一并设计。

### 18.3 T3 — 简介为空时的策略

**阶段性结论**

- **`importWebBook`**：若抓取到的 `introText` 为空，仍插入 **`orderIndex === 0`** 的「简介」条目，并写入**固定占位说明正文**，避免空 `contentText` 导致阅读器异常，并提示用户可回详情页重试或检查登录态。

**仍开放**

- 是否在合规前提下增加「仅拉取 meta/description 的轻量重试」、或与 `AUTH_REQUIRED` 联合的专门引导文案；需评估反爬与超时预算。

### 18.4 T4 — 错误码、三层策略与 UI 文案对齐

**阶段性结论**

- **实现双线并存**：根目录 **`src/content-extractor/*`**（包级算法、测试脚本）与 **`desktop/electron`** 内 **IPC + 页内 `executeJavaScript`** 同时为真；专题 PRD §5 已强调：仅改 **`chapterDetector.ts`** **不会自动**修复桌面「解析目录」行为。
- **对齐基准**：错误码与可观测性以 **`docs/实施-网页导入阅读-Agent执行手册.md`** 为准；主窗口网页采集区的用户可见文案应与 IPC 抛出信息一致（避免同一失败多种说法）。

**仍开放**

- 将桌面侧 `EMPTY_TOC`、`TOC_PARTIAL` 等与 **`总PRD.md` §7.2.7** 错误码表做**逐条对照**，补缺统一命名与「下一步动作」按钮文案。

### 18.5 与专题 PRD 一致的已落地产品行为（摘要）

- 导入前预览与导入后目录：**第 1 条为「简介」**，其后为章节链接行；预览区展示可滚动列表便于验收 TOC。
- 详见专题 PRD §3.3–§4 与 `desktop/electron/db.ts` 中 `importWebBook` 实现。

### 18.6 工程落点速查

| 能力 | 路径 |
